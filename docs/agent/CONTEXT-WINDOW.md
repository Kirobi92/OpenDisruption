## 1. Purpose

The ContextDB is the **only** runtime channel through which the new agents (`opencode`, `openclaw`, `hermes-reasoner`, `obsidian`, `kidi`, `keybrodi`) exchange working state. It is short-lived, zone-tagged, and locally bound.

Long-term knowledge stays in the existing stores (Qdrant collections, Postgres, Obsidian vault on disk). ContextDB is the **transient** middle layer.

---

## 2. Backing store

- **Engine:** Redis (single instance, Phase 1).
- **Bind address:** `${KIROBI_BIND_HOST:-127.0.0.1}:6379` (matches the existing convention in `docker-compose.yml`).
- **Persistence:** AOF every second (so a crash loses Ōēż1 s of context, never all of it).
- **Default TTL:** 24 h on every entry. Callers may override down to 1 h or up to 7 d. Anything longer must be persisted to Qdrant or the Obsidian vault instead.
- **Compose profile:** `kidi`. Redis does not start in the default `docker compose up` path.

No other backing store is permitted. No SQLite fallback, no in-memory dict for production paths.

---

## 3. Key schema

```
ctx:<zone>:<agent>:<uuid>
```

- `<zone>` is one of `PUBLIC`, `WORKSPACE`, `FAMILY_PRIVATE`, `QUARANTINE`, `SACRED`.
- `<agent>` is the writing agent's canonical name (matches `metadata/AGENTREGISTRY.md`).
- `<uuid>` is a v4 UUID generated by the writer.

The zone is part of the **key**, not just the value, so:

1. A wildcard `SCAN ctx:WORKSPACE:*` cannot accidentally surface FAMILY_PRIVATE entries.
2. Zone-scoped readers can use `MATCH ctx:<their max zone>:*` patterns.
3. Misrouted writes are visible at the key level during audits.

---

## 4. Value schema

```json
{
  "agent":      "opencode",
  "zone":       "WORKSPACE",
  "timestamp":  "2026-05-06T17:00:00Z",
  "task_id":    "f0e1...-uuid",
  "payload":    { /* free-form, agent-defined */ },
  "confidence": 0.0,
  "embedding_ref": null,
  "ttl_seconds": 86400,
  "sources":    []
}
```

- `payload` is opaque to ContextDB but must be JSON-serializable.
- `confidence` defaults to `0.0`, meaning **unvalidated / placeholder**. Writers should set a real value (e.g. 1.0 for deterministic outputs, lower for heuristic ones); KIDI uses it as the weight in synthesis.
- `embedding_ref`, when set, points into a Qdrant collection from `metadata/COLLECTION-MAPPING.md` and that collection's zone **must equal** the entry's zone.
- `sources` holds parent ContextDB keys for entries produced by KIDI synthesis.

## 5. Python API surface (Phase 1 target)

```
class ContextDB:
    def put(self, agent: str, zone: Zone, payload: dict,
            *, task_id: str | None = None,
            confidence: float = 0.0,
            ttl_seconds: int = 86400,
            embedding_ref: dict | None = None,
            sources: list[str] | None = None) -> str: ...

    def get(self, key: str, *, requester_max_zone: Zone) -> dict | None: ...

    def query(self, *, requester_max_zone: Zone,
              agent: str | None = None,
              task_id: str | None = None,
              since: str | None = None) -> list[dict]: ...

    def merge(self, keys: list[str], *,
              requester_max_zone: Zone,
              output_zone: Zone) -> str: ...
```

Every read and query takes a `requester_max_zone`. The implementation:

1. Refuses to return any entry whose zone exceeds `requester_max_zone`.
2. Logs every refusal to `kirobi-core/core-events.log` with reason `CONTEXT_ZONE_REJECT`.
3. Refuses `output_zone > min(input zones)` in `merge()`.

There is no `delete_all` method. Deletion is per-key, audited.

---

## 6. Zone enforcement contract

Hard rules for the implementation (Phase 1 must satisfy all):

1. **Write-time zone tagging is required.** A `put` without a `zone` argument raises; there is no default.
2. **Zone is stored in both the key and the value.** Mismatch on read returns an error and logs `CONTEXT_KEY_VALUE_MISMATCH`.
3. **Reads are zone-bounded.** `get`/`query` callers declare their maximum zone; the API never returns entries above it.
4. **No cross-zone iteration.** There is no method that returns entries spanning multiple zones in a single call. Callers must query per zone.
5. **FAMILY_PRIVATE and SACRED writes refuse if `KIROBI_EGRESS_ALLOWED=true`.** This prevents accidental processing on a host with open egress (relevant for the optional Telegram service).
6. **QUARANTINE entries cannot have `embedding_ref`.** Mirrors the rule in `ZONE-POLICY-MATRIX.md`.
7. **SACRED writes require an explicit `human_approval_token` argument.** Mirrors `CLAUDE.md` ┬¦5.

## 7. Telegram boundary

The Telegram service (Phase 5, opt-in) **must** treat ContextDB reads as untrusted: it instantiates `ContextDB` with `requester_max_zone=WORKSPACE` and a separate Redis ACL user with read access only to keys matching `ctx:PUBLIC:*` and `ctx:WORKSPACE:*`. See `TELEGRAM-INTEGRATION.md`.
