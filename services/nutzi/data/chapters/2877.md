# Gateway Adapter Kontext 
Gateway Adapter Kontext 

Im Dialog Gateway Adapter Kontext generieren Sie Adapter für den Datenaustausch. eNVenta stellt die Adapter in diesem Dialog bereit. 

### Gateway Architektur 
Sämtlicher Datenaustausch von und nach eNVenta ERP mit Fremdsystemen wird über die Gateway-Architektur abgewickelt. Pro Datenaustauschszenario ist jeweils ein spezifischer Adapter notwendig (z.B. Lagerverwaltungssystem (LVS) Adapter, EDI Adapter). Diese Adapter sind separat zu beziehen. Alle Adapter werden einheitlich über diesen Menüpunkt verwaltet/konfiguriert. 

Es ist vorgesehen, pro Adapter mehrere Instanzen des Adapters (Adapter Kontext) zu verwenden. 

Beispielszenario: Ein Lagerverwaltungssystem Adapter (LVS) konfiguriert für zwei Kontexte, einmal die Anbindung für das Lager in Stuttgart und die zweite Anbindung für das Lager in Hamburg (jeweils mit unterschiedlichen Einstellungen: Datenbank-Anbindung LVS, Artikelreichweite, …). 

Die Konfiguration erfolgt durch zwei Tabellen, die voneinander abhängig sind. 

Die Event-IDs müssen so vergeben werden, dass die letzte Ziffer der ID für die Art des Events steht: 

- ID "XYZ1" => 1 steht für Insert/Create - ID "XYZ2" => 2 steht für Update - ID "XYZ3" => 3 steht für Delete - ID "XYZ4" => 4 steht für Ignore 
Diese Konvention ist nötig, um die noch nicht verarbeiteten Events vor dem Verarbeiten zu "bereinigen". Gibt es z.B. zu einem Artikel ein unverarbeitetes Insert/Create- und ein Update-Event muss nur eines davon ausgeführt werden. Gibt es dagegen noch zusätzlich ein Delete-Event, muss keines der 3 Events ausgeführt werden. 

Schnittstelle zum Datenimport ins Belegjournal 

Es ist möglich Ausgangsrechnungen, Eingangsrechnungen (alle geprüften Warenrechnungen), Kassenbelege, Eigenbedarfsbelege über eine allgemeine Schnittstelle in eNVenta zu übernehmen. Die Daten werden hierbei in das Belegjournal bzw. in die Kore-Schnittstelle importiert und von dort als Quelle in die Finanzbuchhaltung bzw. Kostenrechnung übernommen. Der Import in die Tabellen kann mittels Job Server auf dem Gateway Adapter Kontext automatisiert werden. 

Die Belege werden in einem definierten Format über die Schnittstelle importiert bzw. exportiert. 

Zur Einrichtung gibt es den Eintrag Adapter-GeneralIO . 

Diesen Eintrag markieren Sie in der Tabelle und öffnen den Dialog Generischer IO Adapter über den Button rechts von der Tabelle. 

Siehe Kapitel Generischer IO Adapter . 

Splitt von Events 

Damit es beim gleichzeitigen Zugriff mehrerer Job Server auf Gateway-Events nicht zu einem Fehler kommt, können einzelne Prozessschritte aufgrund entsprechender Parameter getrennt ausgeführt werden. So lassen sich die einzelnen Prozessschritte eines Gateway-Events besser steuern, in dem ein Job Server den Splitt der Events vornimmt und ein anderer oder mehrere Jobs übernehmen dann den Export der Daten. 

Im Dialog Job-Verwaltung stehen deshalb 3 Parameter zur Verfügung, die einzeln gesetzt werden können. Siehe nachfolgender Screenshot. 

Job-Verwaltung 

Grundsätzlich gilt, dass deren Werte standardmäßig auf 'false' stehen (wegen der Abwärtskompatibilität). 

Die Parameter Keine Ereignisverarbeitung (A.-Kontext-neutral) (FSbool) und Keine Ereignisverarbeitung (A.-Kontext-spezifisch (FSbool) stehen für den Export von Daten zur Verfügung. 

Mit diesen Parametern erfolgt a) der Splitt der Events und b) die Ausführung des gesplitteten Events (der eigentliche Export). 

Mit dem Parameter Kein Aufruf der Hauptmethode des Adapters (TimerElapsed) (FSbool) wird gesteuert, ob der Import über den Adapter gestartet werden soll.
