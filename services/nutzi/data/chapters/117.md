# Job Server Einbindung 

## Allgemein 
Der Job-Server in eNVenta ist der zentrale Service, der automatisiert im Hintergrund Aufgaben übernimmt. 

Die Jobs laufen asynchron über einen separaten Prozess und belasten daher den Sachbearbeiter nicht. 

Bei der Definition des Jobs müssen, aus dem jeweiligen Kontext heraus, alle Daten eingerichtet sein. Bei fehlgeschlagenen Jobs erfolgt keine automatische Information an den auslösenden Sachbearbeiter. Dies muss der Administrator überwachen. 

Es gibt u.a. für folgende Funktionen die Job-Server Funktion: 

- Automatische Lieferzuteilung im Auftrag - Massendruck Verkauf - Bestellvorschlag im Einkauf - Bestellautomat - Reichweitenoptimierung - MIS-Parameter - Webshop-Parameter ->E-Mail-Versand - Intercompany-Adapter - Archivpool - Lieferzuteilung - Datanorm 
Damit ein Vorgang vom Job-Server abgearbeitet werden kann, benötigen Sie als erstes eine Component, die das IcJobServerInterface implementiert. 

Die Job-Verarbeitung erfolgt in 2 Stufen: 

- Job-Schedule: Umfasst einen Auftrag, der alle Informationen für den Job-Server beinhaltet. (Art der Component, Ausführungs-Zeitplan, Parameter) - Job-Instance: Wenn ein Job ausgeführt wird, wird immer eine Job-Instance generiert. An dieser werden auch alle Informationen während der Ausführung gesammelt. Für jede Ausführung gibt es eine Job-Instance. 
## Job-Component anlegen 

Zuerst erzeugen Sie eine Component, z.B. cPurchaseOrderProposalJob. So eine Component sollte immer mit „Job“ enden, damit sie immer als eine Job-Component zu erkennen ist. Diese Component gehört in den entsprechenden Namespace, in den sie inhaltlich passt – also nicht in JobServer. 

Im Namespace fügen Sie eine Referenz auf den JobServer-Namespace hinzu. 
-> Business-Layer -> NV.ERP.Base.JobServer_Interfaces 

Hinterlegen Sie an der Component auf dem General-Register das Interface: 
-> NV.ERP.Base.JobServer.IcJobInterface 

Trennen Sie ggf. mehrere Interfaces durch Komma. 

Dann implementieren Sie die Methoden des Interfaces. 

1.)	ExecuteJob 
In dieser Methode erfolgt die eigentliche Verarbeitung. Sie wird vom Job-Server aufgerufen. 

2.)	GetJobParallelType 
Diese Methode gibt einen Wert zurück, der angibt, ob und wie der Job-Server Jobs mit demselben Typ parallel verarbeiten kann. 

Gültige Rückgabe-Werte 

| mehrfach | Es können beliebig viele Jobs dieses Typs ausgeführt werden. |
| --- | --- |
| 1 pro Benutzer | Pro Benutzer kann nur ein Job dieses Typs ausgeführt werden. |
| einzeln | Es kann nur ein Job dieses Typs ausgeführt werden. |

Die JobInstance besitzt eine ganze Reihe von Methoden, mit der Informationen zurückgeschrieben werden können. Die Informationen werden sofort in die Datenbank gespeichert. Dadurch, dass die Job-Server-Tabellen mit einer separaten DB-Connection arbeiten, erfolgt das Zurückschreiben in einer eigenen kleinen Transaktion – eine evtl. geöffnete eigene Transaktion wird dabei nicht gestört. Innerhalb einer Transaktion können somit Status-Informationen zurückgeschrieben werden. 

Die Methode SetProgress schreibt den Fortschritt in % zurück.

Die AddStateInfo-Methoden schreiben Protokoll-Einträge zurück. So ein Protokoll-Eintrag besteht aus den folgenden Informationen: 

- Typ (Information, Warnung, Error) - Text - Details (Langtext) - Genaue Uhrzeit 
Für Informationen gibt es die AddStateInfo-Methoden. 

Für Warnungen gibt es die AddStateInfoWarning-Methoden. 

Beim Aufruf dieser Methode wird der Status der Job-Instanz automatisch auf „Warnung“ gesetzt. 

Für Fehler (Errors) gibt es die AddStateInfoError-Methoden. 

Beim Aufruf dieser Methode wird der Status der Job-Instanz automatisch auf „Fehler“ gesetzt.

Der Job-Server selber produziert schon einige Status-Meldungen: 

- Job Created - Start ExecuteJob - finished ExecuteJob 
Wenn die Methode ExecuteJob eine Exception wirft, dann wird diese vom Job-Server selber auch in den Status-Meldungen dokumentiert. Der Job wird dann als fehlerhaft gekennzeichnet. 

Es ist also die Entscheidung des Entwicklers, ob die Exception behandelt werden soll und der Job entsprechend kontrolliert beendet werden soll oder ob der Job-Server diese Aufgabe übernehmen soll. Wenn Sie sich selber darum kümmern, wird der Job erfolgreich erledigt. 
## Informationen aktualisieren 
Die Status-Informationen sollten mit Bedacht verwendet werden. Sehr viele und sehr große Einträge können die Performance des Gesamt-Systems sehr negativ beeinflussen. Wenn in einem längeren Job die Anforderung besteht, in regelmäßigen Abständen ein „Lebenszeichen“ zu protokollieren, dann kann die Funktion UpdateJobInfo verwendet werden. Dabei wird der existierende Satz geändert, der Zeitstempel aktualisiert, der Satz an das Ende der Liste verschoben und alle Information (wie bei AddJobInfo) gespeichert. 

Der Funktion UpdateJobInfo wird eine guid übergeben. Diese erhalten Sie mit der Funktion AddJobInfo . UpdateJobInfo gibt selber wieder eine guid zurück. Wenn die übergebene ID nicht gefunden wird, dann wird ein neuer Info-Satz erzeugt und die neue ID zurückgegeben.

## Einbau Job-Server-Button 
Dieser Button ruft den Job-Server-Dialog auf. Diesem muss ein vorkonfiguriertes JobSchedule-Objekt übergeben werden. 

Damit die beiden Logiken in einem „Rutsch“ programmiert werden können, wird die folgende Vorgehensweise empfohlen: 

1.)	Ein Form-Property oJobSchedule (cdJobSchedule) anlegen. 
2.)	Initialisierung in InitObjects 

3.)	Eine Form-Methode, die den JobSchedule mit Daten und Parametern füllt. 

Anmerkung: der Cast nach IcJobInterface ist notwendig, weil die Interface-Implementierungen nur für Components funktionieren und hier mit Interfaces gearbeitet wird. 

4.)	Eine Methode, die ein Job-Schedule entgegennimmt und die Parameter in die Oberfläche überträgt. Diese wird aufgerufen, wenn Sie einen Job aus der Job-Verwaltung bearbeiten möchten.
