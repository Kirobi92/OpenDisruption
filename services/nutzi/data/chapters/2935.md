# EDI Teilnehmer 
EDI Teilnehmer 

Dieser Dialog öffnet sich im Gateway >> Optionen >> EDI >> EDI Teilnehmer . Hier definieren Sie die wichtigsten Grunddaten für jeden Ihrer Teilnehmer, der am EDI-Datenaustausch teilnimmt. 

Voraussetzung: Sie haben im Firmenstamm bereits entsprechende Mandanten angelegt. Dafür ist zwingend die ILN für Lieferanten und Kunden anzulegen! 

Über den Button im Kopfbereich wählen Sie den Kunden oder Lieferanten aus und anschließend im unteren Bereich mit Klick auf den Button das entsprechende Nachrichtenformat. Pro Teilnehmer können mehrere Nachrichtenformate gewählt werden. 

Eine Speicherung ist nur möglich, wenn dem Teilnehmer mindestens ein Nachrichtenformat zugewiesen wurde. 

Sobald Sie in der unteren Tabelle ein Nachrichtenformat markiert haben, erscheinen darunter weitere Optionen (Felder), welche gepflegt sein müssen. Bei einem Export muss so immer das entsprechende Verzeichnis angegeben werden, von wo die Datei gespeichert werden soll. 

Je nach ausgewähltem Nachrichtentyp in der unteren Tabelle (Spalte ‚Nachricht‘) werden unterhalb der Tabellen andere Felder angezeigt und es öffnet sich mit dem Button der passende Dialog. 

Hier muss für die Typen SLSRPT und INVRPT eine Zeile mit einer Lieferantennummer ausgewählt werden. 

Die Verzeichnisse müssen immer vom Broker aus erreicht werden können. 

Der Button neben der unteren Tabelle öffnet den Dialog Nachrichten Auswahl . Hier wählen Sie einen Nachrichtentyp aus und übernehmen ihn entweder mit Doppelklick oder dem Button in den Dialog EDI Teilnehmer . 

Siehe Kapitel Nachrichten Auswahl . 

### Typ SLSRPT 
SLSRPT bedeutet Sales Data Report Message (=Abverkaufsinformationen). Mit diesem Nachrichtentyp können Abverkaufsinformationen an einen Lieferanten gesendet werden. Es wird selektiert nach Lager und Hauptlieferant. 

Wie bekannt können mit dem Export des SLSRPT (=Verkaufsdatenbericht) Abverkaufsinformationen an einen Lieferanten gesendet werden. Dies kann auch über den X-Mapper in den Nachrichtenformaten XML und Text erfolgen. 

Dazu gibt es beim Verkaufsdatenbericht die Möglichkeit, einen Bericht nur für Artikel-Sets zu definieren ( Artikel , Registerkarte Allgemein , Checkbox Artikel-Set ). Ist diese Checkbox aktiviert, werden Artikel mit dem Kennzeichen Artikel-Set für den Bericht berücksichtigt. D.h. für einen Lieferanten kann es mehrere Einträge im Verkaufsdatenbericht geben (mit und ohne Artikel-Set). 

Über den X-Mapper ist ein Zugriff auf die Aufträge (Kopf- und Positionsdaten) und auf den Artikel möglich. Es werden auch die bereits schon vorhandenen Informationen über den X-Mapper ausgegeben bzw. sind über den X-Mapper zugänglich. 
Wenn der Nachrichtentyp SLSRPT aktiviert ist, öffnet sich über den Button neben dem Begriff Art. ausschluss den Dialog Verkaufsdatenbericht . 

Siehe Kapitel Verkaufsdatenbericht . 

### Typ INVRPT 
INVRPT bedeutet Inventory Report (= Lagerbestandsbericht). Mit diesem Nachrichtentyp können Bestandsinformationen für ein Sortiment, das zwischen Kunde und Lieferant vereinbart ist, an den Lieferanten gemeldet werden. Damit kann dieser das Bestandsmanagement übernehmen. Deshalb sind die Bestandsinformationen nach Lager und Hauptlieferant zu exportieren. Dazu ist es zwingend erforderlich den Lagerbestand und die reservierte Menge einzugeben. 

So kann der Lieferant für das vereinbarte Sortiment die Nachbevorratung seines Lagers aufgrund der mit der INVRPT Nachricht übermittelten Informationen vornehmen. 

Bei diesem Vorgang wird im Vorfeld keine Bestellung angelegt oder eine ORDERS Message verschickt! 

Siehe Kapitel Inventory Report . 

### Typ DELFOR 
Für den Nachrichtentyp DELFOR (Kunde, Import) können auch die Message-Formate „XML“ und „TEXT“ ausgewählt werden. Ein entsprechendes X-Mapping und die zu verwendende Prozessregel können ausgewählt werden:

### Nachrichtenformat 
Zur Unterscheidung, ob mit einem EDI Teilnehmer per EDI-Inhouseformat oder XML-Format kommuniziert wird, ist die Spalte Nachrichtenformat editierbar. Hier kann durch Auswahl EDI, XML oder Text (entspricht ASCII oder CSV-Datei) das entsprechende Format gewählt werden. Je nach Auswahl werden die entsprechenden notwendigen Konfigurationsdaten sichtbar. 

Bei einer EDI Kommunikation via XML muss beim Export immer ein XML Mapping angegeben werden. 

Bei einer EDI Kommunikation via XML muss beim Import immer ein XML Mapping und eine Prozessregel angegeben werden. 

Bei Anpassung der Struktur muss darauf geachtet werden, dass diese Struktur für alle EDI-Teilnehmer gilt, welche dieses X-Mapping zugeordnet haben. 

Über die Combobox Prozessregel wird pro Beteiligtem eine Prozessregel festgelest. 

Seit geraumer Zeit nutzt eNVenta die EDI-XML-Lösung. Diese nutzt eine parallele Verarbeitungslogik zum "alten" Inhouse-EDI. 

Die "alte" EDI-Logik wird dann über die nächsten Versionen (zu eNVenta 4.5 …) abgekündigt und durch das X-Mapping ersetzt. 

Die Satzbeschreibung für das EDI-Inhouse-Format für den INVOICE-Export steht auf dem FTP zur Verfügung.

### Feld Dateinamenvorlage 
In diesem Feld können Dateinamen konfiguriert werden. Dafür kann für den EDI-Export je Teilnehmer und Nachricht der zu erzeugende Dateiname mit variablen und statischen Werten definiert werden. Für die variablen Werte werden alle Properties der Export-Komponente (wie im X-Mapper) gemappt. 

Tooltip: 

Die Vorlage für einen Dateinamen ist die Definition eines Dateinamens, der Platzhalter enthalten kann, die zur Ausführungszeit durch den Wert der jeweiligen Variable ersetzt werden. 

Beispiel: 

Vorlage für Dateiname: {DATE}_DOC_{ID}.xml 

Wert der Variable 'DATE' = "20230405" 
Wert der Variable 'ID' = "47110815" 

Endgültiger Dateiname: "20230405_DOC_47110815.xml" 

### Feld Dateiendung 
In diesem Feld können Sie zur Nachrichtenkonfiguration beim EDI Import und Export Dateiendungen frei wählen. 

Die Filterung nach der Dateiendung ist nur im Nachrichtenformat TEXT verfügbar. Die Nachrichtenformate EDI bzw. XML gehen von der jeweils identischen Dateiendung aus. 

Wenn die Angabe der Dateiendung vorhanden ist, werden nur Dateien von diesem Typen für diesen EDI-Teilnehmer und Nachricht importiert. 

Sollte keine Dateiendung angegeben sein, so greift die bisherige Logik. 

### Checkbox Export zuerst als Temp-Datei 
Wenn Sie diese Checkbox aktivieren, wird die Datei zuerst als temporäre Datei gespeichert. So kann die Datei nicht verloren gehen. 

Tooltip: 

Wenn Ja? Beim Export der EDI-Datei wird zuerst eine Temp-Datei geschrieben (.tmp Dateiendung) und diese anschließend umbenannt (.edi Dateiendung). 

Wenn Nein? Die Datei wird direkt mit der .edi Dateiendung exportiert. Dies bringt das Risiko mit sich, dass EDI-Dateien, bei denen der Schreibvorgang noch nicht abgeschlossen wurde, vom Kommunikationspartner schon gelesen werden könnten. 

### Checkbox Datei archivieren 
Wenn Sie diese Checkbox aktivieren, werden Dateien die per EDI erhalten wurden an der entsprechenden Stelle archiviert. So können Sie nachvollziehen, welche Dateien einem Auftrag oder einer Bestellung zugrunde liegt. 

Die Verschlagwortung wird am Objekt über den entsprechenden Dialog vorgenommen (z.B. Auftrag, Bestellung, Rechnungsprüfung). Zusätzlich ist in der Konfiguration der Archivierung beim logischen Archiv auch unbedingt die Checkbox Std. einfügen zu setzen. 

Tooltip:

### Feld X Path-Filter 
Hier können Sie einen XPath-Ausdruck (eine Wahr-/Falsch-Bedingung) hinterlegen, um zu filtern, welche Dokumente überhaupt für diese Konfiguration gültig sind. 

Der Ausdruck wird vor dem Mapping ausgewertet. Wenn die Prüfung nicht wahr (d.h. false) sein sollte, wird nicht gemappt. 

In dem XPath-Ausdruck können Sie z.B. prüfen, ob bestimmte Elemente vorhanden sind, einen bestimmten Wert enthalten und vieles weitere mehr. Die Möglichkeiten sind ähnlich wie bei SQL. 

BEISPIEL 
//BuyerParty/ID='BP1' or //BuyerParty/ILN='4711' 

In diesem Fall muss es in dem Dokument einen XML-Knoten 'BuyerParty' geben, dessen 'ID'-Knoten den Wert 'BP1' oder dessen 'ILN'-Knoten den Wert '4711' enthält. 

HINWEIS 
Der XPath-Ausdruck muss ohne Namespace-Bezüge angegeben werden und wird auch ohne Berücksichtigung der Namespaces im ausgewertet. 

Weitere Dokumentation zur Benutzung von XPath finden Sie unter: 

www.w3.org/TR/xpath
