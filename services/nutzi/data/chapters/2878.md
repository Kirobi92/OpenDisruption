# Generischer IO-Adapter - Einrichtung und Weiterentwicklung 

### Importe & Exporte über Gateway IO-Adapter: Einrichtung und Weiterentwicklung 

#### 1. Grundlagen 
Im eNVenta Modul Gateway gibt es den Gateway-Adapter General IO-Adapter (IO = Input / Output), mit dem Sie unterschiedliche Importe und Exporte durchführen können. 

Im General IO Adapter lassen sich die Exporte und Importe von Daten flexibel konfigurieren, sodass unterschiedliche Daten in unterschiedlichen Formaten und über unterschiedliche Kanäle, d.h. Dateien, Datenbanken usw. realisiert werden. 

Viele dieser Importe und Exporte lassen sich bereits „out-of-the-box“ über eine entsprechende Konfiguration nutzen. Weitere Datenübertragungskanäle (Datei, Datenbank, WebService etc.) können über eine Programmierung hinzugefügt werden. Dazu stellt der IO-Adapter ein entsprechendes Framework bereit, welches im Absatz Eigene Programmierung beschrieben wird. 

Im Folgenden werden die Möglichkeiten von Importen und Exporten des IO-Adapters vorgestellt. 

Exporte 

Sollen Daten aus einem System exportiert werden, stellen sich folgende Fragen: 

- Wann soll exportiert werden, d.h. was ist der Auslöser (Trigger) für den Export? - Benutzer klickt auf den Jetzt exportieren -Button - Zeitgesteuert über einen Job, z.B. alle 10 Minuten oder jeden Tag 7:00 Uhr - Sobald ein bestimmtes Ereignis (Event) auftritt, z.B. Auftragsabschluss - Was soll exportiert werden? - Daten werden von einem Benutzer selektiert - Job-Parameter merken sich die Auswahl der zu exportierenden Daten - Daten sind Event-bezogen, z.B., wenn Auftrag 4711 abgeschlossen wurde, soll dieser exportiert werden - Wohin soll exportiert werden (welches Medium)? - Datei(en) schreiben - Datensätze in eine Datenbank - Aufruf eines Web-Services - Per E-Mail, usw. - Wie sollen die Daten exportiert werden (d.h. welches Format)? - XML-Format - CSV - Textformat mit fixen Positionen / Längen der Datenfelder - Bei Export in andere Datenbank: welche Daten sollen in welche Tabelle / Spalte eingefügt werden? - Soll der Export dokumentiert werden, d.h. sollen die exportierten Daten nach dem Export verändert werden, z.B. ein Export-Datum setzen oder ähnliches. 
Auslöser (Trigger) 

Da der IO-Adapter ein Gateway Adapter ist, wird der Export asynchron durch ein Event (Ereignis) ausgelöst. Tritt das Trigger-Event auf, wird zunächst nur ein Event-Datensatz in die Gw_EventExport Tabelle geschrieben. Dieser Datensatz enthält die Referenz zum Quell-Datensatz und die Info. um was für ein Event es sich handelt. 

Erst wenn zu einem späteren Zeitpunkt (manuell durch Benutzer oder per Zeit-gesteuertem Job) der IO-Adapter ausgeführt wird, führt das Event zu einem Daten-Export. 

In eNVenta gibt es zwei Arten von Events, die im Gateway verarbeitet werden können: 

- Programmierte Events 
(Welche Arten von Events es gibt und wann sie auftreten wird programmatisch festgelegt -> Änderungen müssen programmatisch gemacht werden) - Konfigurierte Events (siehe Event Management ) 
Events lassen sich ohne Programmierung und individuell konfigurieren. 
Bisher lassen sich im IO-Adapter nur die konfigurierbaren Events nutzen. Es ist jedoch geplant auch die Nutzbarkeit der programmierten Events zu implementieren. 

Laden der Daten für den Export 

Ein wesentlicher Teil der Konfiguration ist es die Daten, die exportiert werden sollen, bereitzustellen, d.h. diese zu laden / zu filtern / zu sortieren und diese evtl. zu standardisieren. 

Die Ausgangsdaten für den Export ist der vom konfiguriertem Event referenzierte Datensatz. Dieser kann im X-Mapping-Editor genutzt werden, um Daten daraus in das gewünschte Format zu übertragen. 

Es besteht aber auch die Möglichkeit anhand der Daten aus dem Event-Datensatz ein anderes Datenobjekt zu laden, welches dann als Datengrundlage für den Export (d.h. X-Mapping-Editor) genutzt wird. 

Beispiel: 
Das konfigurierte Event „Auftragsposition Lieferfreigabe“ bezieht sich auf eine einzelne Auftragsposition (Komponente: cdSalesOrderDetail). 

Wird eine IO-Adapter Export-Konfiguration auf dieses Event eingerichtet, kann entweder der Datensatz der Auftragsposition als Datengrundlage für den Export genutzt werden oder Sie könnten anhand der Daten dieses Datensatzes z.B. den ganzen Auftrag laden und diesen als Export-Daten nutzen.

Auslöser Event 

In der Event-Combobox werden alle Events angezeigt, die im Gateway Kontext aktiviert wurden. Aus dem gewählten Event ergibt sich der Typ des Quelldatenobjekts. In dem Beispiel ist es cdPurchaseOrderMaster, welches einen Datensatz der Tabelle BestellKopf repräsentiert. 

Dieses Datensatzobjekt steht für den weiteren Export zur Verfügung und kann entweder direkt über den X-Mapper genutzt werden oder zuvor über eine Transformation in ein anderes Datenobjekt umgewandelt werden. 

Auslöser ohne Event 

Auslöser ohne Event 

Der Export im IO-Adapter lässt sich auch ohne ein Event auslösen. In diesem Fall wird bei jedem Durchlauf des IO-Adapters der Export durchgeführt. Wann der Export erfolgt, wird somit durch die Zeitsteuerung des Jobs bestimmt. 

Das Datenobjekt, welches für die weitere Verarbeitung genutzt wird ist dabei vom Typ cdGeneralIOConfigExp , welches dem aktuellen Konfigurationsdatensatz entspricht. 

In der Regel wird anschließend eine Transformation genutzt.

Eine X-Mapping-Konfiguration unterscheidet sich grundsätzlich durch den Typ der Datenquelle, in diesem Fall die Komponente „cPOExport“, und dem Typ des Ausgangsobjektes, in diesem Fall ein XML-String.

Generisches IO Adapter Telegramm erstellen 

Mit dieser Funktion kann mit Hilfe des Ausdruckseditors ein String-Wert generiert werden, der beim Export des Vorgangs in die Datenbank-Tabelle „GeneralIOSentTelegrams“ geschrieben wird. 

Auf diese Weise kann gespeichert werden, ob ein bestimmter Export bereits erfolgt ist, in dem man an den entsprechenden Vorgängen eine Logik programmiert, die in der o.g. Tabelle nach entsprechenden Einträgen sucht. 

Hinweis: Da diese Funktion derzeit nur dann einen Mehrwert schafft, wenn Sie zusätzliche Logik implementieren, raten wir von der Benutzung ab. 

Weil Sie hier nicht ohne Programmieren weiterkommen, empfehlen wir, beim verwendeten Datenexport-Objekt das Interface IcSentTelegramsInterface zu implementieren. Dieses definiert die Methode SetSentTelegramsInfo() , welche nach dem Export aufgerufen wird. Hier kann dann auf beliebige Weise der Export gespeichert werden z.B. Statusänderung, Export-Datum setzen und so weiter. 

Import 

Anders als beim Export werden die Import-Vorgänge nicht durch aufgetretene Events ausgelöst, sondern die Verarbeitung erfolgt bei jedem Durchlauf des IO-Adapters. Ob bei jedem Durchlauf auch Daten importiert werden, hängt von der Datenquelle (neue Import-Daten müssen vorhanden sein) und der Konfiguration ab. 

Import-Kanal 

Beim Import wird für alle Import-Konfigurationen geltend einmalig der Import-Kanal ausgewählt. 

Import-Kanal 

Wie auch beim Export-Kanal lassen sich die Kanal-spezifischen Einstellungen mit dem -Button vornehmen. 

Es folgen die vorhandenen Import-Kanäle. 

Generischer Dateiimport 

Mit diesem Kanal lassen sich Dateien aus einem Import-Verzeichnis importieren. Verarbeitete Dateien können in ein Ziel- bzw. im Fehlerfall in ein Fehler-Verzeichnis verschoben werden. 

Dateien lassen sich über ihren Namen per regulärem Ausdruck filtern und auch über den Namen steuern, wie die Datei verarbeitet wird – d.h. welcher Prozess ausgeführt wird. 

PTV Datenimport 

Hierfür gibt es ein gesonderte Anleitungsdokument: Anbindung an PTV . 

Kennung 

Die Kennung dient dazu, die unterschiedlichen Typen der Datensätze oder Dateien zu unterscheiden. Dadurch lässt sich je nach Typ des Datensatzes (Satzart) ein eigener Mapping-Vorgang und ein eigener Prozess ausführen, also eine eigene Import-Konfiguration verwenden. 

Innerhalb eines IO-Adapter Gateway-Kontext muss eine Kennung eindeutig sein - sie darf also nur einmal vergeben werden. Welche Kennungen möglich sind, ist abhängig vom verwendeten Import-Kanal und/oder der angebundenen Datenquelle. 

Wird der generische Datei-Import verwendet, kann die Kennung innerhalb des Dateinamens gesucht und dadurch die Import-Konfiguration, d.h. das X-Mapping und die Prozess-Einstellungen, ermittelt werden. Dazu muss der entsprechende Parameter für den Export-Kanal gesetzt werden. Wird die Kennung nicht verwendet, um Dateien zu identifizieren, erfolgt die Identifizierung über das X-Mapping. Die Kennung kann dafür beliebig vergeben werden - muss aber trotzdem eindeutig bleiben. 

Prozess / Regel / X-Mapping-Konfiguration 

Prozessregel in der X-Mapping-Konfiguration 

Über den gewählten Prozess wird gesteuert, was mit dem Import-Datensatz / Datei passieren, also welcher Vorgang ausgeführt werden soll. Je nach gewähltem Prozess muss eine entsprechende Prozessregel und eine passende X-Mapping Konfiguration gewählt werden. 

Die Prozessregel-Auswahl ergibt sich allein aus dem gewählten Prozess. Beim X-Mapping ist nicht nur der Prozess entscheidend, welche Konfigurationen in Frage kommen, sondern auch das Input-Objekt, welches vom gewählten Import-Kanal ausgegeben wird. Dieses kann wiederum z.B. von den Dateien abhängen, die importiert werden.

Daten nach dem Export modifizieren 

Oft besteht die Notwendigkeit nach dem Export die exportierten Daten zu modifizieren, z.B. den Status auf exportiert zu setzen oder ähnliches (siehe auch Absatz Generisches IO Adapter Telegramm erstellen ). 

Das entsprechende Export-Datenobjekt muss dazu das Interface NV.ERP.Gateway.Adapter.AdapterGeneralIO.IcSentTelegramsInterface implementieren.
