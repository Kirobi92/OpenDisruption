# Scanner Filtered List 

## Grundlagen 
Das cfrmScannerList dient zur Darstellung von Listenwerten und der Auswahl einzelner Einträge zur Bearbeitung. Dies ist das geeignete Basisform, wenn die Anzahl der Einträge überschaubar ist. Enthält die Ergebnisliste viele Einträge, so ist es sinnvoll eine Liste mit Filterkriterien zu verwenden. 

Dazu haben wir das Basisform cfrmScannerFiltered List geschaffen. Diese leitet vom cfrmScannerList ab und erweitert die Mechanismen von dort, um die Logik zum Filtern. 

Dazu erweitert die Logik die Ansicht um eine Liste der Filterkriterien, in welcher Sie durch Anwählen den Filter wieder löschen können. 

## Filtereinträge 
Die Filtereinträge bestehen aus einer Collection vom Typ cFilterCriteriaGroup. 
Jede dieser Gruppen enthält eine beliebige Anzahl von Filterkriterien. 

Ein Filterkriterium besteht immer aus 3 Werten: 
Key: der Propertyname auf welchen gefiltert werden soll 
Value: Der Wert mit welchem das Property geprüft werden soll 
sType: Eine Klartextbezeichnung für die Anzeige in der Liste. 

Die Filterwerte werden automatisch im Restore-Mechanismus verarbeitet. 

## Eingabewerte 
Anders als im cfrmScannerBase oder cfrmScannerList wird hier der Eingabewert von der Basisklasse schon gesetzt. 
Aufgrund der unterschiedlichen Werte, die bei einer Filterung meistens erwartet werden, ist dieser standardmäßig ein String. 
Das Label und der Vorschlagswert können über die Methoden GetEnteredValueLabel ( Muss überschrieben werden) und GetEnteredValueInfo gesetzt werden. 

## Eingabewerte der Liste zufügen 
In der Methode GetFilterGroups kann der eingegebene Wert gegen die unterschiedlichen Filtermöglichkeiten verprobt werden, welche erwartet werden. 
Wird eine Eingabe erfolgreich als Kriterium identifiziert, so wird diese der Ergebnisliste zugeteilt. 
Enthält diese mehr als einen Eintrag, so öffnet sich eine Auswahlliste, in welcher der Benutzer entscheiden kann, welches Kriterium er jetzt zufügen möchte. 

## Suche bzw. Aktualisierung der Liste 
Da wir hier mit größeren Datenmengen und häufigeren Aktualisierungen rechnen, haben wir hier nicht einfach nur einen Load Mechanismus eingebaut. 
Gibt es bereits ein Filterkriterium, so wird beim Zufügen eines weiteren Filterkriteriums eine Filterung der Ergebnisliste über LINQ durchgeführt. 
Nur wenn sich die Filterkriterien, mit welchen die Werte geladen wurden, verändert werden, wird neu aus der DB geladen. 
Dies ist alles in der Basisklasse realisiert und muss nicht extra angesteuert werden. 

## Auswahl, wenn es nur ein Ergebnis gibt 
Enthält das Ergebnis genau einen Eintrag, können Sie diesen automatisch auswählen und ein Workflow Event starten. 

Dazu muss man 2 Methoden beachten: 
- SetSingleResult 
In dieser kann der gefundene Eintrag in ein Selected Property der Wahl gesetzt werden. - StartSelectIfOnlyOneEntryExists 
Hierüber kann gesteuert werden, ob direkt ein Folgeprozess ausgelöst werden soll, wenn es nur einen Eintrag gibt. Der Standardwert ist true .
