# Anbindung an PTV 

### 1. Definition 
Die Software PTV Route Optimiser dient der Tourenplanung und –optimierung und ist mittels einer Schnittstelle an eNVenta angebunden. Es handelt sich hierbei um eine Datenbank-Schnittstelle, d.h. eNVenta schreibt die Daten direkt in eine PTV-Datenbank und liest sich auch die Antworten aus dieser Datenbank ein. 

Die Schnittstelle zur Datenbank ist definiert und es wird auch eine vorgefertigte Definition mit ausgeliefert. Grundsätzlich bietet die Definition eine Grundlage für diese Schnittstelle. Die Schnittstelle muss für jede Integration geprüft und konfiguriert werden. Hierzu ist aber eine entsprechende Abstimmung mit PTV notwendig, da ggf. auch auf Seiten PTV Anpassungen vorgenommen werden müssen, um mit anders interpretierten Daten umgehen zu können. 

Auf Seiten von eNVenta werden folgende Daten standardmäßig exportiert: 

- Stammdaten - Kundenanschrift - Lieferantenanschrift - Versandanschriften - Aufträge - Packstückinformation - Retouren zur Abholung - Bestellungen zur Abholung - Angaben zur technischen Entladung 
Der Export der Stammdaten erfolgt im Regelfall zeitgesteuert, damit die Datenpflege nicht doppelt erfolgen muss. Die Stammdaten werden hierbei in eNVenta gepflegt und an PTV übergeben. Innerhalb von PTV können diese Daten veredelt bzw. ergänzt werden. Ebenso müssen weitere Stammdaten (Fahrzeuge, Lagerorte, …) in PTV als Stammdaten erfasst werden. Ein Rückimport der Daten nach eNVenta ist nicht vorgesehen. 

Für die Planung der Touren werden von eNVenta die einzutourenden Vorgänge übergeben. Der Zeitpunkt der Übergabe der Daten an PTV kann über das Eventmanagement definiert werden. Die Übergabe kann zu einem sehr frühen Zeitpunkt, wie z.B. Auftragsabschluss erfolgen, als auch später, z.B. bei der Fakturafreigabe. Jedes Vorgehen hat seine Vor- und Nachteile und muss im Projekt entsprechend definiert werden. 

In eNVenta ist eine Vorplanung der Touren möglich (Tourenplanung in eNVenta vorausgesetzt) und diese werden als Information an PTV übergeben. PTV plant die Touren aber eigenständig neu und das Ergebnis der Planung in PTV wird in die Tourenplanung von eNVenta importiert. 

Über die Protokollierung ist es möglich zu sehen, welche Daten zwischen den Systemen ausgetauscht wurden. 

### 2. Technische Voraussetzungen 
Notwendige Lizenzen in eNVenta 

- Versand oder Logistik - Gateway - PTV 
Version eNVenta 

Die Mindestversion für die Nutzung der Schnittstelle zu PTV ist eNVenta 4.2 . 

Datenbanken 

PTV nutzt eine eigene Datenbank für den Austausch der Daten. Die Information, welche Datenbanken unterstützt werden, erfragen Sie bitte direkt bei PTV. Für die Anbindung an eNVenta gelten die gleichen Softwarevoraussetzungen wie für den Betrieb von eNVenta . 

### 3. Grundlagen 
Der Export der Daten erfolgt über den General IO Adapter . In diesem werden die verschiedenen Export-Definitionen mittels X-Mapping hinterlegt und können ggf. angepasst werden. 

Um den Zeitpunkt des Exports zu definieren, wird das in eNVenta integrierte Event Management verwendet. Für das Event Management werden die notwendigen Events und Aktionen mitgeliefert und können ggf. angepasst werden. Auch können eigene Events registriert werden, um den Export an PTV zu steuern. 

### 4. Einrichtung PTV 

#### Einrichtung Framework Studio 
Für die Kommunikation zwischen eNVenta und der PTV-Datenbank ist diese zwingend als Eintrag in der Connection Group „cnPTV“ in Framework Studio beim Publish der Anwendung anzugeben. 

Run Wizard 

#### Einrichtung in eNVenta 
Die notwendigen Daten werden als Ressource ausgeliefert. Diese werden bei der Veröffentlichung in den Broker geschrieben: 
- Unterordner ResGeneralIO_Adapter - Datei PTV_Standard.xml 
Diese Datei ist unbedingt manuell in einem Verzeichnis zu speichern, welches mittels Server Datei Manager von eNVenta aus erreichbar ist. 

Gateway Adapter Kontext 

Im Dialog Gateway Adapter Kontext (über Navigator Werkzeuge >> Gateway zu öffnen) können die Grunddaten von PTV importiert werden. Hierzu klicken Sie im Menü Optionen auf den Eintrag Kontext Import/Export . 

Adapter Kontext Import/Export 

Über den Bereich Import auf der rechten Seite können die Grunddaten von PTV importiert werden. Geben Sie hierzu den Pfad der Datei an und klicken anschließend auf den Button Analyse . Jeder zu importierende Adapter muss analysiert werden, da der Import als Basis für weitere Schnittstellen dienen kann. 

Nach der Analyse wird automatisch der Adapter ermittelt und Sie können den Import dann entsprechend konfigurieren. Klicken Sie hierzu auf den Button Konfigurieren . 

Prozessregeln 

Im Dialog Prozessregeln entscheiden Sie, welche Daten Sie aus der mitgelieferten Konfiguration importieren möchten. 

In der allgemeinen Konfiguration entscheiden Sie, ob ein bereits bestehender Adapter ersetzt werden soll. Bei einem Erst-Import wird die Angabe mit null belassen. Nur wenn Sie eine Konfiguration komplett überschreiben möchten, ist es notwendig den zu überschreibenden Adapter auszuwählen. 

Achtung: Auch andere Gateway-Adapter, wie z.B. EDI können damit überschrieben werden. 

Über die Option Events importieren werden die notwendigen Eventmanagement Events importiert. 

Über die Adapter-spezifischen Konfigurationen werden die Mappings für die einzelnen PTV-Telegramme importiert. Diese sind fest vorgegeben und können ggf. im Nachgang überarbeitet werden. 

Die Option Process Settings steuert die Prozessregeln für die Verarbeitung der Rückantworten von PTV an eNVenta . 

Die Option Transformers importieren steuert die Transformationen innerhalb der X-Mapping-Vorlagen für den Export zu PTV. 

Default-mäßig sind alle Optionen gesetzt, um den Erst-Import schnell zu ermöglichen. Nach dem Schließen des Dialogs kann im Dialog Adapter Kontext Import/Export der Import mittels des Buttons Import gestartet werden. 

Nach dem Import der Daten kann der General IO Adapter geöffnet werden. Hierzu markieren Sie den Gateway-Adapter und klicken auf den Button auf der rechten Seite der Tabelle. 

Generischer IO Adapter 

Im Dialog Generischer IO Adapter können Sie nun die Details der PTV-Schnittstelle einstellen bzw. bearbeiten. 

Als Importobjekt muss PTV Datenimport ausgewählt sein. 

Die Einstellung Zielprotokoll ist für die Protokollierung der Schnittstelle verantwortlich (siehe Absatz Protokollierung der Schnittstelle ). 

Generischer IO Adapter Registerkarte Export 

Über die Registerkarte Export steuern Sie den Zeitpunkt und den Inhalt des Exports von eNVenta zu PTV. 

Wenn Sie einen Eintrag markieren bzw. hinzufügen, können Sie die Optionen hinterlegen: 

Event: Angabe des Events aus dem Event Management, wann die Daten exportiert werden bzw. zur Versendung im Gateway registriert werden. 

Transformer: Hinterlegung der Umwandlung vom Datenobjekt eNVenta zum Prozessobjekt. Mittels Button neben dem Feld Transformation öffnet sich der Dialog Konfiguration Transformer . 

Konfiguration Teilnehmer 

Diese Objekte sind fest vorgegeben. Eine Anpassung der Objekte bedarf einer größeren Umstellung und ggf. Programmierung und sollte daher nicht vorgenommen werden. 

Mittels der Ziel-Methode werden die notwendigen Daten geladen. Die Übergabe-Parameter werden hierbei in der unteren Tabelle hinterlegt. Die Auswahl erfolgt über den TreeView daneben. 

Über den X-Mapper erfolgt die Auswahl des Mappings zwischen dem Prozessdatenobjekt in eNVenta zur Objekt-Struktur in PTV. Über den Button kann der X-Mapping Editor geöffnet werden, um das Mapping zu prüfen oder zu ändern. 

X-Mapping Editor 

Die Struktur auf der rechten Seite entspricht dem Datenobjekt aus der PTV-Datenbank. Eine Änderung der Datenbank kann nur durch PTV vorgenommen werden und muss dann in eNVenta entsprechend eingelesen werden. 

Exportobjekt: Angabe, dass der Export in einer FS Komponente erfolgen muss. Da der Export immer in die PTV-Datenbank geht, muss immer der Wert Export in FS Komponente angegeben werden. 

Export Wert: Er gibt die Bedingung an, ob das Mapping verwendet werden soll. Bei einem Event können mehrere Übergaben parallel gestartet werden, doch nicht immer sollen alle ausgeführt werden. 

Option Ohne Event: Option darf für PTV nicht aktiv sein. 

Option Transformer aktiv: Option muss für Übergabe an PTV immer aktiv sein. 

Generischer IO Adapter Registerkarte Import 

Identifier: Hier ist der fixe Wert „TOUR“ anzugeben. Dies dient der Interpretation für den Import der Daten aus der Übergabe-Datenbank. 

Über den X-Mapper wählen Sie die Definition, um die Daten aus der PTV-Datenbank in das eNVenta Objekt umzuwandeln. 

Über die Option Prozessobjekt geben Sie an, welche Prozessregeln für den Import angewendet werden sollen. In den Prozessregeln definieren Sie: 

- Anlage neuer Touren - Wenn die bereits vorhandenen Touren nicht ausreichend sind, werden ggf. neue Touren angelegt. - Abschluss Touren - Die bisher angelegten Touren werden nach dem Import geschlossen. Eine weitere Eintourung ist danach nur manuell möglich. - Entfernung von Vorgängen aus der Tour, wenn diese nicht zuvor an PTV übergeben wurden (Nachläufer) - Belege, welche nicht ursprünglich an PTV übergeben und dort in der Planung berücksichtigt wurden, werden aus der Tour entfernt. In diesem Falle übernimmt PTV die komplette Tourenplanung. - Übergabe der Daten an V-Log-Schnittstelle für Paketversand - Wenn PTV entscheidet einen Beleg per Paketdienst zu versenden, so kann - die Schnittstelle V-Log vorausgesetzt - der Vorgang automatisch an V-Log gemeldet werden. 

### Protokollierung der Schnittstelle 
Die Protokollierung der Schnittstelle erfolgt mittels der globalen Protokollierung und der integrierten Lösung log4net . Die Protokollierung kann wahlweise in eine Textdatei, Datenbank, per SMTP oder Wiedervorlage erfolgen. Ebenso können Sie die verschiedenen Ebenen des Logs einstellen, um z.B. Informationen in die Datei zu schreiben und nur bei Fehlern eine automatische E-Mail versenden lassen. 

Für die Einrichtung der Protokollierung öffnen Sie die Protokoll-Konfiguration im Navigator unter Menü System >> Administration >> Protokollierung >> Konfiguration-Protokoll . 

Konfiguration-Protokoll 

Auf der linken Seite sehen Sie die vorhandenen Protokoll-Quellen, welche definiert werden können. Markieren Sie das Protokoll PTV und klicken anschließend auf den Button auf der rechten Seite, um einen neuen Appender (Empfänger) hinzuzufügen. 

Geben Sie dem Appender eine Bezeichnung und wählen Sie in der Spalte ‚Typ‘ aus, wie Sie die Daten protokollieren möchten. Je Typ gibt es unterschiedliche Konfigurationsmöglichkeiten. Zur Auswahl stehen folgende Typen: 

- eNVenta Datenbank - Textdatei - SMTP - Wiedervorlage 
Für die Generierung des Logs wird ein Caching verwendet, damit nicht andauernde Zugriffe auf die Protokollierungen notwendig sind. Das Logging können Sie hierbei im Dialog Parameter Allgemein festlegen. 

Parameter Allgemein 

Hier geben Sie im Feld Logging Refresh an, nach wie vielen Minuten jeweils der Cache in das Ziel-Log übernommen wird. Bei der Angabe von leer bzw. 0 erfolgt das Logging ohne Cache. Spätestens mit dem Beenden von eNVenta wird der Cache geleert und ins Log geschrieben. 

### Protokollierung in der Datenbank 
Bei Auswahl der Datenbank zur Protokollierung werden alle Daten direkt in die Datenbank geschrieben. 

Achtung: Das Wachstum der Datenbank wird hierbei nicht begrenzt. Eine Löschung des Protokolls ist nicht über die Oberfläche möglich. Nur über den Zeithorizont kann eingestellt werden, wie lange die Daten vorhanden bleiben. 

Geben Sie die Log-Stufe an, welche Daten Sie protokollieren möchten. Folgende Log-Stufen stehen zur Verfügung: 

- Debug: Angabe wird nur geschrieben, wenn eNVenta im Debug-Modus gestartet wurde (für Entwicklungen und Fehlersuche). - Info: Es werden alle Informationen mitgeschrieben. - Warn: Es werden nur Warnungen und Fehler mitgeschrieben. - Error: Es werden nur Fehlermeldungen mitgeschrieben. - Fatal: Es werden keine Fehlermeldungen mitgeschrieben. Eintrag erfolgt nur bei Systemabsturz oder bei nicht Erreichbarkeit der Gegenstelle. 
Geben Sie den Zeithorizont zur Datenspeicherung an. Die Angabe erfolgt hierbei in Tagen. Sobald die Anzahl Tage überschritten ist, werden die älteren Daten in der Datenbank automatisch gelöscht. 

Zur Betrachtung der Protokollierung der Daten gibt es den Dialog Protokoll Datenbank . Diesen öffnen Sie über den Navigator Menü System >> Administration >> Protokollierung >> Protokoll Datenbank . 

Protokoll Datenbank 

### Protokollierung in einer Textdatei 
Bei der Protokollierung als Textdatei werden die Daten in einer Datei anstatt in der Datenbank gespeichert. Der Vorteil ist: die Datenhaltung erfolgt außerhalb der Datenbank und ist nur durch den vorhandenen Festplattenplatz begrenzt. 

Für die Speicherung der Daten müssen Sie einen Pfad angeben. Dieser Pfad muss von allen Brokern und Services (insbesonders Jobserver) aus erreichbar sein. 

Bei der Protokollierung in eine Textdatei haben Sie folgende Optionen: 

- XML-Protokoll: Speicherung der Daten als XML-Datei. Wenn nicht aktiviert, erfolgt die Speicherung in einer txt-Datei. - Per Datum: Pro Tag wird eine neue Datei generiert. - Per Größe: Alternative Steuerung zum Datum. Wenn das Protokoll eine Dateigröße überschreitet, wird eine neue Log-Datei erzeugt. Die Definition der Größe erfolgt im Feld Max. Größe KB . - Max. Anzahl Dateien: Gibt die maximale Anzahl an Dateien an. Wird diese Grenze überschritten, werden die ältesten Dateien im Pfad automatisch gelöscht. 

### Protokollierung SMTP 
Bei der Protokollierung SMTP erfolgt ein automatischer Versand der Nachricht als E-Mail. Hierzu müssen Sie die entsprechende Konfiguration des SMTP-Servers angeben. Sie können angeben, welcher Absender angezeigt werden soll sowie den Empfänger der E-Mail. Zusätzlich hinterlegen Sie hier den Betreff der E-Mail. Der Text wird anhand der Log-Stufe automatisch generiert. 

### Protokollierung Wiedervorlage 
Bei der Protokollierung mittels Wiedervorlage wird jede Meldung an den hinterlegten Sachbearbeiter gesendet. 

- Die Log-Stufen Debug und Info haben die Priorität normal . - Die Log-Stufe Warning hat die Priorität wichtig . - Die Log-Stufen Error und Fatal haben die Priorität sehr wichtig .
