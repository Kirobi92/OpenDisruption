# VK-Positionen bearbeiten Registerkarte Kalkulation 
Position Registerkarte Kalkulation 

Auf dieser Registerkarte können Werte für die Kalkulation der Position geändert werden. 

Die Auftragsart steuert, ob der Wert im Feld Posbetrag inkl. MwSt. oder exkl. MwSt. angezeigt wird. 

Über den Shortcut <Strg + F8> im Auftrag gelangen Sie ebenfalls direkt in die Registerkarte Kalkulation . 

Auf dieser Registerkarte können Sie Einträge vornehmen und die Veränderungen beobachten, ohne die Daten tatsächlich in die Position zu übernehmen. Erst wenn Sie mit dem Button den Dialog verlassen und die nachfolgende Abfrage mit Ja bestätigen, werden die neuen Daten in die Auftragsposition gespeichert. 

Beginnend mit dem Einkaufspreis werden unter Berücksichtigung der Nebenkosten und Rabatte der Verkaufspreis, der Ertrag und die Spanne berechnet. 

Die EK-Kalkulation erfolgt auf bis zu 7 Nachkommastellen. Dafür haben die DB-Felder APos.LEK und APos.Nebenkosten 25 Dezimalstellen. Angezeigt werden aber im Normalfall weiterhin nur 2 Nachkommastellen. In einem möglichen Grenzfall werden alle signifikanten NK-Stellen angezeigt. 

Sie können direkt in die Kalkulation eingreifen und die Werte verändern. Es erfolgt sofort eine Neukalkulation des gesamten Schemas. So können beispielsweise die Spanne, der Einkaufspreis, die Rabatte oder der Verkaufspreis manuell verändert werden. 

Wird die Handelsspanne angepasst, so wird auch der Verkaufspreis angepasst. Hierbei werden die %-Rabatte, die absoluten Rabatte und der MTZ korrekt berücksichtigt. 

Ist der Auftrag bereits fakturiert, kann das Kalkulationsschema nicht mehr verändert werden! 

Die Kalkulation erfolgt immer in der Währung des Auftrags. 

Bei der Fakturierung einer Handelsstückliste wird der Lager-GEK korrekt als EK in die Auftragsposition übernommen, unabhängig vom Dispo-Kennzeichen in der Stückliste. 

Für die Kalkulation eines Artikel-Sets steht Ihnen ein weiteres Menü zur Verfügung. 

Beim aktiven Lieferschein-Identitätspreisverfahren wird immer der GEK als Kalkulationsbasis genommen. Ebenso wird dieser vor dem Lieferscheindruck neu ermittelt und aktualisiert in der Auftragsposition gespeichert. 

Von dieser Regel sind Diversartikel ausgeschlossen, da diese zu differenziert sind, um eine saubere Ermittlung zu ermöglichen. 

Rundung der Werte für die Positionskalkulation 

Für die Rundung der Positionsbeträge gelten in eNVenta feste Regeln. 

- Aufgrund gesetzlicher Anforderungen ist es notwendig, dass sich der Positionsbetrag centgenau aus allen anderen ausgewiesenen Beträgen der Rechnung ergeben. - Gleichzeitig besteht teilweise die Anforderung prozentuale Rabatte monetär als Beträge und nicht als Prozentwerte auf der Rechnung darzustellen. - Der Positionsbetrag wird auf zwei Nachkommastellen gerundet. 
Aus den Punkten 1. bis 3. ergibt sich, dass auch die monetären Rabattbeträge zwischengerundet werden müssen. Dies muss bei der Berechnung des rabattierten Betrags beachtet werden. Die prozentualen Rabatte können nicht ohne Zwischenrundung angewandt werden. 

Schematische eNVenta Kalkulationsvariante 

Gegeben: Betrag und prozentualer Rabatt x1, monetärer Rabatt x2. 
Dann rechnet eNVenta den rabattierten Betrag folgendermaßen aus: 

Betrag = VK / VKPro * Menge, 
Rabattierter Betrag = (Betrag - Betrag * x1 gerundet - x2) gerundet 

Und nicht: 
Rabattierter Betrag = (Betrag - Betrag * x1 - x2) gerundet 

Begründung dieser Variante 
In der Rechnung sollen alle Positionsrabatte MONETÄR darstellbar sein und der Positionsbetrag auf der Rechnung muss sich über Betrag +/- aller (monetär ausgewiesener Positionsrabatte) CENTGENAU ergeben (aufgrund der gesetzlichen Anforderungen). 

Deshalb muss zwischengerundet werden. 

Beispiele 
Beispiel 1: 
Gegeben ist eine Auftragsposition mit folgenden Werten: 
VK = 10,00€, VKpro = 1, Menge = 10 to, Rab1 = 5%, Rab2 = 10%, Rab3 = 15% 

Rohumsatz = VK / VKpro x Menge = 100,00€ 

Monetärer Äquivalentrabatt1 = Rohumsatz x 5/100 = 100,00€ x 5/100 = 5,00€ => gerundet 5,00€ 
Zwischenwert1 = Rohumsatz - Äquivalentrabatt1 = 95,00€ 
Monetärer Äquivalentrabatt2 = Zwischenwert1 x 10/100 = 95,00€ x 10/100 = 9,50€ => gerundet 9,50€ 
Zwischenwert2 = Zwischenwert1 - Äquivalentrabatt2 = 95,00€ - 9,50€ = 85,50€ 
Monetärer Äquivalentrabatt3 = Zwischenwert2 x 15/100 = 85,50€ x 15/100 = 12,825€ => gerundet 12,83€ 
Zwischenwert3 = Zwischenwert2 - Äquivalentrabatt3 = 85,50€ - 12,83€ = 72,67€ 
PosBetrag = Zwischenwert3 = 72,67€ => gerundet 72,67€ 

Beispiel 2: 
Gegeben ist eine Auftragsposition mit folgenden Werten: 
VK = 10,00€, VKpro = 1, Menge = 10 to, Rab1 = 11,01%, Rab2 = 10,00€ 

Rohumsatz = VK / VKpro x Menge = 100,00€ 
Monetärer Äquivalentrabatt1 = Rohumsatz x 11,01/100 = 100,00€ x 11,01/100 = 11,01€ => gerundet 11,01€ 
Zwischenwert1 = Rohumsatz - Äquivalentrabatt1 = 100,00€ - 11,01€ = 88,99€ => gerundet 88,99€ 
Monetärer Rabatt2 = 10,00€ 
Zwischenwert2 = Zwischenwert1 - Monetärer Rabatt2 = 88,99€ - 10,00€ = 78,99€ 
PosBetrag = Zwischenwert2 = 78,99€ => gerundet 78,99€ 

Differenzierung automatischer und manuell gesetzter Rabatte und Zuschläge 

Eine Unterscheidung zwischen manuell erfassten und automatisch generierten, aber manuell angepassten Rabatten/Zuschlägen ist deutlich zu erkennen. Dafür steht die Spalte ‚Manuell‘ auf dieser Registerkarte zur Verfügung. 

D.h. im Fall einer Neukalkulation können manuell erfasste bzw. geänderte Rabatte/Zuschläge von automatisch gezogenen Rabatten/Zuschlägen unterschieden werden. Dadurch erfolgt die Differenzierung in eNVenta eindeutig, da es die unterschiedlichsten Arten von Rabatten und Zuschlägen gibt. Z.B. aus der Rabattverwaltung Verkauf und Einkauf, Materialteuerungszuschläge und kunden- bzw. artikelspezifische Rabatte. 

Zusammenfassung: eNVenta erkennt in jedem Fall, ob 
- es sich um einen automatisiert gefundenen unveränderten Rabatt/Zuschlag handelt. - es sich um einen automatisiert gefundenen manuell geänderten Rabatt/Zuschlag handelt. - es sich um einen manuell erfassten Rabatt/Zuschlag handelt. 
Hinweis: Trifft eine Vielzahl an Rabatten/Zuschlägen gleichzeitig zu, so bildet eNVenta im Rabatt 6 einen Multirabatt aus den verbliebenen zutreffenden Rabatten/Zuschlägen. Dieser Multirabatt erhält ebenfalls eine eindeutige Kennung (Buchstabe C). Bei einer Neukalkulation kann eNVenta die zugrundeliegenden weiteren Rabatte/Zuschläge nicht erkennen.

### Checkbox Kein Rabatt 
Wenn an der Auftragsposition das Kennzeichen Kein Rabatt gesetzt ist, so zieht der Zuschlag / Rabatt im Auftrag nicht, unabhängig von der Einstellung im Artikelstamm. 

Wenn das Kennzeichen Kein Rabatt an der Auftragsposition gesetzt ist, kommt eine Hinweismeldung, wenn Rabatte oder Zuschläge verändert werden. 

Wenn das Kennzeichen an der Auftragsposition nicht gesetzt ist, so muss berücksichtigt werden, ob die Kondition (z.B. die Preisgruppe) vorgibt, dass keine Rabatte gezogen werden sollen. 

Konkret wird das Kennzeichen Kein VK Rabatt in den Preisgruppen auch für die Basispreisgruppen 1 bis 3 berücksichtigt und das Kennzeichen Kein Rabatt am Artikel wird für die Preisgruppe 0 mitberücksichtigt. 

Beispiel: 
Ein ein- oder zweistelliger Rabatt / Zuschlag wird zum Artikel definiert. 
Wenn an der Auftragsposition Kein Rabatt gesetzt ist, so zieht der Zuschlag / Rabatt im Auftrag nicht, unabhängig von der Einstellung im Artikelstamm. 

Im Artikelstamm ist das Kennzeichen Kein Rabatt gesetzt. 
Die Position wird erfasst -> das Kennzeichen gesetzt. Speichern. 
Im Artikelstamm das Kennzeichen deaktivieren. 
Neuladen des Auftrags und Neukalkulieren. 
=> Das Kennzeichen bleibt aktiviert. 

In Artikelstamm ist das Kennzeichen Kein Rabatt nicht gesetzt. 
Das Kennzeichen wird manuell aktiviert. Speichern. 
Neukalkulieren -> das Kennzeichen BLEIBT aktiviert. 
Rabatte werden nicht gezogen. 

Tooltip: 

Bei Erfassung einer Position wird die Checkbox gesetzt, falls die gleichnamige Checkbox im Artikelstamm 
gesetzt ist. Kann manuell übersteuert werden. 
Wenn gesetzt, so werden keine Rabatte oder Zuschläge aus den Konditionen gezogen. 

### Checkbox Kein Kopf-Rabatt 
Wenn Sie diese Checkbox aktivieren, wird die selektierte Auftragsposition von der Rabattierung auf Auftragskopfebene ausgeschlossen. 

Ist diese Checkbox bereits gesetzt, kann sie manuell deaktiviert werden. 

Dieser Parameter wird an folgenden Stellen vorbelegt: 

- Initial aus der Preisgruppe, wenn die Position neu ist. - Dialog Kassieren : bei Rückkehr aus der Gutschriftsauswahl , Belegauswahl oder OP Suche . - Bei der Erstellung einer Gutschrift aus der Retoure heraus, wenn der Betrag 'Gutschrift' an der Retourenposition gesetzt ist. 
Bei der manuellen Neukalkulation im Auftrag wird die Kennung "Kein Kopfrabatt" nicht neu ermittelt. Beim Kopieren des Auftrags in einen neuen Beleg wird die Kennung jedoch aktualisiert. 

Wird bei der Generierung einer Gutschrift aus einer Retoure der Positionswert (über den Dialog Retourenposition , Feld Betrag Gutsch ) manuell eingetragen, wird diese Checkbox automatisch aktiviert. D.h. Kopfrabatte werden in der Gutschrift nicht berücksichtigt! 

Tooltip: 

Wenn gesetzt, so werden keine Kopfrabatte angewandt.
